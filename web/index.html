> WHAT IS WHOIS DEEP

Standard WHOIS (registry-level) queries the TLD registry — ICANN, Verisign, NomiNet — and often returns redacted contacts behind privacy shields. That single hop leaves two entire data layers untouched.

whois-deep goes two levels deeper:

  • Registrar WHOIS — queries the registrar's own WHOIS server directly (not the registry's). Registrar servers frequently expose billing contacts, reseller chain, and nameserver provisioning info that the registry-level response omits.
  • IP WHOIS via RIR — resolves the domain's A record, then queries the responsible Regional Internet Registry (ARIN, RIPE, APNIC, LACNIC, or AFRINIC) directly. Returns ASN, organisation name, netblock, CIDR notation, block size, and hosting provider classification.

Together these two extra hops answer: who registered it, who bills them, what network carries it, and who physically operates that infrastructure.

> WHY THIS SCRIPT

Raw whois output is unstructured, inconsistent text. Different registrars return different field names, different line endings, and sometimes plain garbage. whois-deep parses both registrar WHOIS and IP WHOIS responses into a consistent schema.

Beyond parsing it also:

  • Classifies the hosting environment — CDN edge node, shared hosting pool, dedicated VPS, cloud hyperscaler, or known CDN provider — based on org name and netblock patterns.
  • Calculates actual block size from the inetnum/CIDR so you know if you are looking at a /8 cloud superblock or a /27 dedicated allocation.
  • Caches results in a local SQLite database with a configurable TTL (default 24 h) to avoid hammering RIR rate limits on repeat queries.
  • Emits structured JSON, making output pipeline-friendly for jq, grep, and downstream scripts.

Pairs with whois-extracter.py (registry-level parsing) for complete three-layer WHOIS coverage.

> HOW IT WORKS
INPUT: --domain target.com
↓
cache.db (24h TTL)
↓ (stale or --no-cache)
dig +short A → resolve IP
↓
whois <ip> → RIR query
whois -h <registrar-whois-server> <domain>
↓
parse: ASN · netblock · org · contacts
↓
JSON output + stdout summary
> USAGE

Basic domain query:

# query registrar WHOIS + IP WHOIS, cache result
python3 whois-deep.py -d target.com

Write structured JSON to file:

python3 whois-deep.py -d target.com --output results/target.json

Bypass cache (force fresh query):

# --no-cache: ignore cached result, re-query RIR and registrar WHOIS
python3 whois-deep.py -d target.com --no-cache

Custom cache TTL:

# --ttl: seconds before a cached entry is considered stale (default 86400)
python3 whois-deep.py -d target.com --ttl 3600

Submit to RIR Board (public index):

# --submit: opens a GitHub Issue with your result JSON
# CI validates the JSON and merges it into the index
python3 whois-deep.py -d target.com --submit

First-run configuration (RIR Board token):

# --reconfigure: re-run the interactive setup wizard
python3 whois-deep.py --reconfigure

> HOSTING SIGNATURES

whois-deep matches 14 hosting fingerprints against the RIR org name and netblock. Each classification carries a different pentest implication:

  • Cloudflare — origin IP hidden behind proxy; real server is elsewhere. Do not scan this IP for open ports — you are looking at Cloudflare's edge, not the target.
  • AWS — EC2 or AWS-managed service. Note the region from the CIDR; check for exposed S3, ALB, or ECS metadata endpoints.
  • GCP — Google Cloud compute. Similar to AWS; look for GCP metadata server at 169.254.169.254.
  • Azure — Microsoft cloud. IMDS endpoint accessible on unprotected instances.
  • DigitalOcean — common for small VPS deployments; check default SSH keypair provisioning and droplet metadata.
  • Vultr — budget VPS; no WAF by default; often misconfigured SMB or Redis exposed.
  • Linode / Akamai — Linode was acquired by Akamai; netblocks may appear under either org. Treat as VPS — direct access, no edge proxy.
  • Fastly — CDN similar to Cloudflare; origin IP hidden. Check for Host header bypass to reach origin directly.
  • Akamai CDN — enterprise CDN; origin IP strictly hidden; Akamai-specific headers (X-Check-Cacheable, X-Akamai-*) in responses.
  • Hostinger — shared hosting; likely cPanel; check for phpinfo, backup exposure, and weak FTP credentials.
  • HostGator — shared cPanel hosting; high co-tenancy; directory listing and phpinfo common.
  • PDR (PublicDomainRegistry) — registrar-side hosting; low-cost shared environment; limited WAF.
  • Hetzner — German VPS/dedicated; direct exposure; common for self-hosted stacks; check default Nginx/Apache configs.
  • Unknown — no pattern matched; inspect org_name and net_name manually; consider dedicated or colocation environment.
